## 1. Claude Code CLI 개요 및 설치
Claude Code는 코드베이스를 읽고, 파일을 수정하며, 터미널 명령을 실행하는 **에이전트(Agent) 코딩 툴**입니다. 단순한 질의응답을 넘어 상태를 읽고, 판단하고, 실행하는 반복적인 워크플로우를 터미널 내에서 직접 수행합니다.

* **초기 설정:** 프로젝트 루트 디렉터리(`cd [프로젝트 폴더]`)에서 실행하며, 최초 사용 시 인증(Login) 절차가 필요합니다.

## 2. Claude Code 아키텍처 (4계층)
Claude Code를 안정적으로 사용하려면 단일 프롬프트에 의존하기보다, 병목이 발생하는 레이어를 파악하고 보강해야 합니다.

| 레이어(Layer) | 주요 역할 및 구성 요소 | 문제 발생 시 증상 | 보강 방법 |
| :--- | :--- | :--- | :--- |
| **1. 규칙 (Rule)** | `claude.md`, 금지 사항, 코딩 스타일 등 세션 간 유지되어야 하는 기준 | 세션마다 결과의 품질이나 스타일이 들쭉날쭉함 | 프로젝트 규칙(`claude.md`) 구체화 |
| **2. 컨텍스트 (Context)** | 해결할 문제, 관련 파일, 실패 로그 등 현재 작업의 문맥 | 모델이 엉뚱한 파일을 건드리거나 방향을 잃음 | 작업/파일 범위를 명확히 좁혀서 제시 |
| **3. 전략 (Strategy)** | 계획 수립, 서브 에이전트 분리, 자동화/검토 여부 판단 | 수정 품질이 떨어지거나 작업 범위가 걷잡을 수 없이 커짐 | 조사/수정/검토 세션을 분리 |
| **4. 실행 및 검증 (Execution)** | 파일 수정, 테스트 실행, PR 생성, 자동 검증(Hooks/Actions) | 코드는 맞는데 의존성 충돌이나 런타임 버그가 발생함 | Hooks, CI/CD, 자동화 스크립트 연동 |

## 3. 대규모 코드베이스에서의 컨텍스트 컨트롤
규모가 큰 프로젝트에서 Claude가 헤매는 이유는 모델의 성능 부족이 아니라 **컨텍스트의 경계가 흐리기 때문**입니다. 정보를 '많이' 주는 것보다 '정확하게' 주는 것이 핵심입니다.

### 컨텍스트 컨트롤 3단계
1.  **문제 범위 축소:** 무엇을 고치는지 한 문장으로 정의하기.
2.  **파일 범위 축소:** 전체 코드를 맡기지 말고, 관련 파일 후보군만 먼저 모은 뒤 실제 수정 파일을 좁히기.
3.  **세션 범위 축소:** '조사 세션', '수정 세션', '검토 세션'을 섞지 않고 분리하기. 세션이 너무 길어지면 요약 후 새로 시작하는 것이 유리함.

### 에이전트 친화적 코드베이스의 조건
AI 에이전트가 코드를 오해하지 않으려면 시스템 자체가 탄탄해야 합니다.
* 충분한 테스트 커버리지 확보
* README(문서)와 실제 코드 구조의 일관성
* 명확한 빌드 및 실행 방식
* 흔들리지 않는 코드 스타일과 설계 규칙

## 4. 고급 작업 모드: Plan, Thinking, Sub-agents
복잡한 문제일수록 즉시 코드를 수정하는 것보다 계획과 역할을 나누는 것이 실패 비용을 줄입니다.

* **Plan Mode (플랜 모드):** 코드를 수정하지 않고(Read-only) 코드베이스를 분석하여 계획만 세우는 모드.
    * *사용 시기:* 영향 범위를 확신할 수 없을 때, 여러 디렉터리가 얽혀 있을 때, 테스트가 약한 레거시 코드를 다룰 때.
* **Sub-agents (서브 에이전트):** 작업을 섞지 않기 위한 구조화 장치 (Explore, Plan, General 등).
    * *사용 원칙:* 역할 경계(조사용, 수정용, 검토용 등)가 분명할 때만 생성. 애매한 상태에서 에이전트만 늘리면 혼란이 가중됨.
* **Thinking Mode (사유 모드):** 즉각적인 실행보다 '판단의 품질'이 중요한 불확실성 높은 문제에 연산력을 집중하는 모드. (단순 로그 파싱 등에는 불필요)

## 5. Skills vs Tool Use (MCP)
반복되는 워크플로우와 외부 시스템 연동을 구분하여 설계해야 과설계나 실행 실패를 막을 수 있습니다.

| 구분 | Skills (스킬) | Tool Use / MCP (도구 사용) |
| :--- | :--- | :--- |
| **정의** | 클로드가 **'어떻게 일할지(How)'**에 대한 재사용 가능한 작업 절차 | 클로드가 **'무엇으로 실행할지(What)'**에 대한 외부 시스템 확장 |
| **주요 목적** | 반복되는 절차나 지시를 매번 설명하지 않도록 템플릿화 | 모델 혼자 할 수 없는 외부 데이터 조회나 시스템 제어 |
| **적용 사례** | 로그 요약 루틴, PR 초안 작성, 배포 전 체크리스트 점검 등 | Jira 이슈 조회, 외부 DB 접근, 사내 시스템 상태 확인 등 |
| **판단 기준** | 같은 절차를 세 번 이상 반복해서 설명하고 있을 때 | 클로드가 방법은 아는데 외부 접근 권한/기능이 없어 멈출 때 |

> **추천 흐름:** 외부 명령이 필요 없는 반복 작업은 우선 **Skills**로 표준화합니다. 이후 외부 데이터나 병렬 실행이 반드시 필요한 경우에만 **Tool Use**를 붙여서 작게 시작해 점진적으로 확장하는 것이 안정적입니다.
