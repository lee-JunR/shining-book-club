

# CLAUDE.md 구현 및 관리 전략

> 💡 **핵심 명제**
> **"CLAUDE.md는 Claude에게 프로젝트를 설명하는 파일이지, 모든 위험을 막아주는 보안 장치가 아니다."**
> 지침의 길이가 짧을수록 Claude가 더 높은 정확도로 제안을 준수하므로, 핵심 위주로 압축하여 작성하는 것이 좋습니다.

---

## 1. CLAUDE.md의 주요 역할

* **프로젝트 메모리(Project Memory) 역할**: 프로젝트 전반의 컨텍스트를 유지하는 기억 장치로 기능하며, 팀원들과 소스 제어(Git 등)를 통해 투명하게 공유됩니다.
* **반복 제거**: 매번 대화할 때마다 설명해야 하는 반복적인 컨텍스트를 한곳에서 관리합니다.
* **아키텍처 경계 설정**: 프로젝트의 구조적 규칙과 경계를 명시하여 일관성을 유지합니다.
* **검증 기준 제공**: 코드 작성 및 프로젝트 빌드 시 준수해야 할 테스트 및 검증 명령을 제공합니다.

---

## 2. CLAUDE.md의 계층 구조 (Layers)

| 계층 분리 | 관리 경로 및 도구 | 주요 포함 내용 |
| --- | --- | --- |
| **Enterprise / Managed** | 조직/시스템 관리자 제어 | 조직 보안 정책, 회사 표준 규격, 관리자 전역 제어 정책 |
| **User** | `~/.claude/CLAUDE.md` | 개인 선호도, 개인적으로 자주 사용하는 명령 및 매크로 |
| **Project** | 프로젝트 루트 (`/CLAUDE.md`) | 팀 공통 규칙, 아키텍처 경계 설정, 빌드/테스트 명령 |
| **Local** | `CLAUDE.local.md` (`.gitignore` 필수) | 개인 로컬 개발 환경 맞춤 설정, 실험적 지침, 공유 불가 메모 |

* **Auto Memory 작동 특성**
* 프로젝트 시작 시 **첫 200줄 또는 25kb 중 작은 쪽**만 로드되므로 핵심 지침은 상단에 간결하게 배치해야 합니다.
* 해당 Git 저장소의 모든 워크트리(worktree)가 이 메모리를 공유합니다.



---

## 3. CLAUDE.md vs settings.json 역할 분리

민감한 파일이나 경로에 대한 접근 제한은 `CLAUDE.md`에만 선언적으로 적어두는 것이 아니라, `settings.json`의 보안 정책을 통해 **이중으로 강력하게 강제**해야 합니다.

| 구분 | CLAUDE.md | settings.json / permissions |
| --- | --- | --- |
| **성격** | 가이드라인 · 판단 기준 · 규칙 설명 | 기술적 강제 · 허용/차단 정책 |
| **예시** | *“Do not read .env file...”* | `"deny": ["Read(./.env)"]` |
| **통제 효과** | **소프트(Soft) 차단**<br>

<br>(Claude가 지침을 이해하고 자발적으로 따름) | **하드(Hard) 차단**<br>

<br>(시스템 레벨에서 실제 접근 및 실행을 차단) |
| **공유 방식** | 소스 제어(Source Control)에 포함<br>

<br>→ **팀 전체 공통 공유** | `.claude/settings.json` → **팀 공유**<br>

<br>`.claude/settings.local.json` → **개인 (`.gitignore`)** |
| **사용 원칙** | 반복되는 판단 기준, 아키텍처 설명, 작업 스타일 정의 | **반드시 막아야 하는 명령**, **민감 파일 접근 제어** |
