# 컨텍스트 컨트롤이란 무엇인가

1. 문제 범위를 줄이는 것
2. 파일 범위를 줄이는 것
3. 세션 범위를 줄이는 것

AI를 잘 쓰는 사람은 에이전트를 많이 쓰는 사람이 아니라 에이전트끼리 인터페이싱 할 때 어떤 것을 넘겨줄지 컨트롤하는 사람 즉 컨텍스트 컨트롤을 잘 하는 사람이다.

이 뿐 아니라 코드베이스 자체가 에이전트 친화적인 것도 중요하다.

에이전트 친화적으로 코드베이스를 유지하는것도 중요.
 - 테스트 커버리지가 충분한가
 - README와 실제 코드베이스스가 일관적인가
 - 설계 패턴이 프로젝트 전체에서 비슷하게 유지되는가
 - 빌드와 실행 방식이 명확한가
 - 스타일과 규칙이 흔들리지 않는가

그래서 에이전트 네이티브 엔지니어는 단순히 도구를 많이 붙이는 것이 아니라 설계와 테스트, 그리고 빌드의 핵심을 탄탄하게 만든 뒤 그 위에 에이전트를 얹는게 맞다.

항상 단계적으로 하는 게 맞다.

-> 그래서 새로운 도구와 워크플로를 얼마나 빨리 흡수하고 구조화할 수 있는지가 중요한 능력이 될것


많이 주는것보다 정확히 주는것이 중요.


# Plan First, Thingking Mode, Subagents란 무엇인가

## Plan First
1. 엔트로픽 공식 문서에서는 Plan Mode란 Read Only로 코드베이스를 분석해서 계획을 세우는 모드라고 설명
 -> 복잡한 변경, 코드베이스 탐색, 안전한 리뷰에 적합
2. 해당 모드는 범위를 좁히고 계획을 세우는 것이 바로 플랜모드다.

### 실무에서는 어떻게 플랜모드를 사용해야하는가
 1. 영향 범위를 확신할 수 없을 때
 2. 여러 디렉토리, 서비스가 얽혀있을 때
 3. 테스트가 약하거나 레거시 코드라 수정 한번이 연쇄 영향을 만들 수 잇을 때
 -> 좋은 플랜 first는 범위를 좁히고 계획을 세우는 것이다.

## Sub Agent - 작업을 섞지 않게 만드는 구조화 장치 (작업을 쪼개는 이유 : 메인 컨텍스트를 퓨어하게 만들기 위해.)
1. explore
2. plan
3. general purpose
4. custom subagent

- 역할이 분명할때만 에이전트를 만든다. -> 조사용 수정용 검토용처럼.
- 작업 경계가 모호한데 만들면 오히려 혼란만 생김


## Thinking Mode 
- 즉시 실행보다 판단이 더 중요한 작업에 계산 자원을 더 쓰는 전략.
- 용도 : 넓은 수정, 설계 대안 비교, 원인 후보가 여러개인 디버깅에 유리
- 반대로 로그 한줄 파싱, 파일 경로 찾기에는 과함.
-> 불확실성이 클때만 쓰는 모드.

## 요약.
1. 문제가 단순하고 범위가 명확하면 바로 실행
2. 문제가 복잡하고 범위가 넓으면 plan mode
3. 작업 유형이 서로 다르면 subagent로 나눔
4. 결과 품질이 중요하지만 범위는 아직 애매하면 thinking mode



# Skills 와 Tool Use 란 무엇인가

## Skills : 클로드 코드에게 어떻게 일할지를 재사용하는 것
 - 매번 같은 종류의 로그를 분석
 - 같은 형식의 PR 설명
 - 같은 체크리스트로 배포 전 점검
## Tool Use : 무엇으로 실행할지
 - 외부 시스템을 조회
 - 파일 시스템 외의 다른 실행 주제와 연결
 - 병렬 도구의 호출

 ## 판단 기준
 - 반복되는 프로세스 + 외부 절차 필요 없음 -> Skills
 - 외부 시스템 호출, 데이터 조회, 병렬 실행 - tool use, mcp
