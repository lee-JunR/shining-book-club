GUI로 AI를 사용하면 코드만 생성하기 때문에 대부분의 절차를 사용자가 직접 검수할 필요가 있다.
그렇기에 해당 검수 작업을 과정으로 만들어서 CLI에 지정하고, 해당 절차를 반영한 코드의 결과를 낼 수 있는 것이 주 목적이다.

CLI AI는 작업을 이어서 진행한다는 특징이 있다.


# 전체 구조

##규칙
 - 클로드.md, 프로젝트의 기본 규칙(금지, 코딩스타일)등의 기본 축을 담당한다.

##컨텍스트
 - 문제에 대한 이해와 해결, 제대로 문제에 맞는 파일만을 찾는지, 실패한 로그를 분석하거나 수정범위를 지정하는 등의 유지보수 과정을 담당한다.

##전략
 - 수정하는 코딩의 정합성이나 구조를 담당한다.
 * 수정에 대한 계획 범위를 나누는 것은 코드 분석에 대한 결과범위에 따라 달라진다.
    - 문제가 단순할 경우 explore
    - 문제가 복잡하게 이어져 여러 파일의 수정을 거치는 등의 범위가 넓은 경우 plan-frist
    - 작업 유형이 다 방면으로 필요할 경우 subagent
    - 품질이 중요하지만 범위가 크지 않은 경우 thinking mode
##실행/검증
  - 실제로 전략에 대한 적용 및 해당 결과에 대한 테스트. 완성한 결과물에 대한 뒷 작업을 담당한다.

* 각 담당 내용부분을 나누어 해당 과정속에 문제에 대한 수정을 하기 쉽도록 관리한다.

* 해당 관리의 세부 디테일을 지정해야 큰 틀이 제대로 완성된다.


# 과정을 4분할 한 이유

수정을 하기전에 계획을 먼저 만들지 않고, 바로 수정을 하게 될 경우. 잘못된 파일이나 코드를 수정하는 상황이 발생했을 때 재보수 또는 다른 오류가 연쇄적으로 발생할 가능성이 높다.
그렇기 때문에, READ ONLY 기준으로 먼저 수정 계획을 분석(컨텍스트) 후 수정계획(전략)을 세우는과정을 거친다.
