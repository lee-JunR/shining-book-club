## 1. claude.md 구현 및 하네스 엔지니어링 환경 설정

### claude.md의 역할과 구성
* **목적:** 프로젝트 규칙, 코딩 스타일, 금지 사항을 클로드 코드에 명시적으로 고정
* **기능:** 반복 작업 제거, 아키텍처 경계 설정, 테스트 명령 및 완료 기준 제시
* **작성 원칙:** 프로젝트 지침을 중심으로 짧고 명확한 문서 형태로 작성
* **역할 분리 (md vs 환경 설정):**
   * **claude.md:** 규칙과 판단 기준을 고정하는 가이드 문서 역할
   * **하네스 엔지니어링(Harness Engineering):** 세팅을 통해 민감 파일이나 건드려선 안 되는 금지 파일에 대한 접근을 강제로 차단하는 구조적 안전망(Guardrail) 역할 수행

### 메모리 계층 구조 (4단계)
| 계층 | 설명 |
| :--- | :--- |
| **엔터프라이즈** | 조직 전체의 보안 정책 및 회사 표준 지침 |
| **유저** | 개발자 개인의 선호도 및 코딩 스타일 |
| **프로젝트** | 해당 팀의 공통 프로젝트 지침 |
| **로컬** | 개인 실험 및 임시 메모용 |

### 오토 메모리와 로컬 메모리 관리
* **오토 메모리:** 같은 Git 저장소 내 모든 워크 트리에서 내용이 공유됨
* **로컬 메모리:** 워크 트리별로 분리되어 독립적인 실험용으로 활용 가능
* **주의사항:** 메모리 격리가 완벽하지 않으므로, 개발자가 상태를 명시적으로 통제해야 함

## 2. 클로드 코드 세션 오염 관리

### 세션 오염의 원인
* 대화가 길어짐에 따라 불필요한 컨텍스트가 누적되어 초기 설정(`claude.md`)의 제약 사항이 희석됨
* 여러 작업의 맥락이 섞이면서 클로드 코드가 규칙을 잊거나 금지 파일을 건드리는 현상 발생

### 세션 관리 핵심 루틴 (명령어 활용)
* **`/clear` (초기화):** 대화 컨텍스트를 완전히 초기화하여 작업 단위를 명확히 분리
* **`/compact` (압축):** 대화를 요약하여 핵심 정보만 유지
   * *팁:* 압축 시 변경 파일 목록, 실행 명령, 테스트 결과 등 **요약에 포함할 항목을 명확히 지시**해야 정보 유실을 막을 수 있음
* **규칙 승격:** 요약 갱신을 통해 새롭게 도출된 중요한 규칙이나 결정 사항은 `claude.md`나 규칙 파일로 승격시켜 영구 적용

## 3. 컨텍스트 재설계 (명확한 프롬프팅)

### 엉뚱한 수정이 발생하는 이유
* AI 모델 자체의 문제보다는 **지시와 검증 기준이 불명확한 경우**가 대부분
* 수정 범위, 금지 범위, 검증 기준이 모호하면 클로드 코드가 불필요하게 넓은 범위의 파일을 건드림

### 재설계 4단계
1. **문제 재정의:** 해결해야 할 문제를 한 문장으로 명확히 정의
2. **범위 명시:** 수정 가능 범위와 **수정 금지 범위**를 분리하여 명확히 지시
3. **조건 설정:** 절대 깨지면 안 되는 핵심 조건(Test case 등) 명시
4. **검증 기준:** 작업 완료 판단 기준과 구체적인 확인 방법 제시

### 실무 적용 팁
* **큰 작업:** 바로 코드를 수정하라고 지시하지 말고, **작업 계획(Plan)부터 제시**하도록 요구
* **작은 작업:** 과도한 지시 없이 간단하고 빠르게 처리
* **오염 방지:** 이미 꼬인 세션에 설명을 계속 덧붙이면 상황이 악화되므로, `/clear` 후 재작업 권장

## 4. 이슈부터 PR까지의 자동화 흐름

### 클로드 코드를 활용한 자동화 단계
1. 이슈(Issue) 내용 파악 및 분석
2. 작업용 브랜치 자동 생성
3. 초안 코드 작성 및 커밋 생성
4. Draft PR(초안 풀 리퀘스트) 생성
> **⚠️ 필수 사항:** 코드 최종 리뷰, CI 파이프라인 통과 확인, Merge 승인은 반드시 **사람(개발자)의 검토**를 거쳐야 합니다.

### 자동화 적용 가이드
* **적합한 대상:** 반복적이고 유사한 패턴을 가진 작업
* **부적합한 대상:** 권한 구조가 미정리되었거나, 리뷰 문화가 부재한 보안 민감 환경
* **한계:** 자동화가 끝까지 완벽한 해결책이 될 수는 없으며, 초기 아키텍처 설계와 결과물 검토에 사람의 시간 투자가 반드시 필요함

## 5. 병렬 워크 트리(Worktree) 운영 및 충돌 해결

### 병렬 세션과 워크 트리의 차이
* **동일 디렉터리 내 세션:** 대화는 분리되지만 파일 시스템을 공유하므로, 병렬 작업 시 파일 충돌 및 테스트 꼬임 발생 가능
* **워크 트리 격리:** 각 Git 워크 트리는 독립된 파일 시스템, 브랜치, 작업 내역을 가지므로 클로드 코드를 완벽히 독립적으로 실행 가능

### 워크 트리 운영 전략
* **독립 세션 생성:** 브랜치별로 독립된 워크 트리를 구성해 클로드 코드 병렬 실행
* **불필요 파일 제외:** `.gitignore` 등을 활용해 클로드 코드가 추적할 필요 없는 파일 사전 차단
* **종료 관리:** 작업 완료 후 해당 워크 트리와 세션을 안전하게 제거

### 충돌 발생 시 해결 절차
1. **직접 수정 지시 금지:** 병합 충돌 시 클로드 코드에게 "바로 코드를 고쳐"라고 명령하지 않음
2. **원인 분석 요구:** 충돌 원인과 양쪽 브랜치의 변경 내용을 먼저 설명하도록 지시
3. **계획 수립:** 어떻게 병합할 것인지 구체적인 계획을 세우도록 요구
4. **수정 및 테스트:** 합의된 병합 계획에 따라 코드를 수정하고 단위 테스트 진행
> 클로드 코드가 임의로 코드를 덮어쓰지 않도록, 충돌 원인 분석 ➔ 계획 수립 ➔ 수정의 순서를 엄수하는 것이 핵심입니다.
