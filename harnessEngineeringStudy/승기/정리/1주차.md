# AI 스터디 1주차 - Context Control 정리

## 1. Context Control이란?

이번 주 키워드는 한 줄로 요약하면  
**“많이 주는 것보다, 딱 필요한 것만 주는 것”**이라고 느꼈다.

Claude Code에서 컨텍스트 컨트롤은 크게 세 가지를 줄이는 거라고 이해했다.

1. 문제 범위 줄이기  
   - 한 번에 전부 해결하려 하지 말고, 이번 턴에 풀 문제를 더 잘게 쪼개기.

2. 파일 범위 줄이기  
   - 관련 없는 디렉터리·파일은 과감히 빼고, 진짜 관련 있는 부분만 보여주기.

3. 세션 범위 줄이기  
   - 중간 로그·실험 결과를 다 들고 가지 말고, 필요한 결론만 요약해서 메인 대화에 남기기.

결국 에이전트를 많이 쓰는 사람보다, **에이전트가 볼 정보의 양과 범위를 잘 다루는 사람이 더 잘 쓴다**는 메시지로 느껴졌다.

---

## 2. 에이전트 친화적인 코드베이스

강의에서 반복해서 나온 포인트가  
“AI를 잘 쓰려면, 코드베이스부터 에이전트 친화적으로 만들어야 한다”였다.

개인적으로 정리한 체크리스트는 이 정도다.

- 테스트 커버리지가 어느 정도는 있다.  
- README와 실제 코드 구조가 크게 다르지 않다.  
- 프로젝트 전반에 설계 패턴·스타일이 일관된다.  
- 빌드/실행/테스트 방법이 문서나 스크립트로 정리돼 있다.

이게 되어 있어야, 에이전트에게 **“많이 보여주는 대신, 정확히 필요한 것만 보여주기”**가 가능해진다고 느꼈다.

## 3. Plan Mode / Subagent / Thinking Mode

이 세 가지는 “언제 바로 실행하지 말고 한 번 더 생각할 것인가?”에 대한 옵션이라고 이해했다.

- Plan Mode  
  - 읽기 전용으로 코드베이스를 탐색하면서, 먼저 영향 범위와 계획부터 세우는 모드. 
  - 레거시 코드나 영향 범위가 넓은 수정에는 바로 수정보다 이걸 먼저 쓰는 게 안전하다.

- Subagent  
  - 리서치용, 설계용, 구현용 같이 **작업 종류를 분리해서** 메인 대화를 깨끗하게 유지하는 장치.
  - 역할이 애매하면 굳이 안 만들고, 조사/수정/검토처럼 딱 떨어질 때만 쓰는 게 좋다.

- Thinking Mode  
  - 바로 실행보다 “판단 품질”이 더 중요한 문제에, 계산 자원을 더 쓰는 모드.
  - 단순 검색·경로 찾기에는 과하고, 설계 비교나 원인 후보가 많은 디버깅에 어울린다.

내가 정리한 사용 기준은 이렇다.

1. 문제가 단순하고 범위가 명확하면 → 바로 실행  
2. 문제가 복잡하고 영향 범위가 넓으면 → Plan Mode  
3. 작업 타입이 서로 다르면 → Subagent로 분리  
4. 결과 품질은 중요한데, 아직 상황이 애매하면 → Thinking Mode

## 4. Skills vs Tool Use (짧게)

강의에서 잡힌 구분은 간단히 말하면 이거였다.

- **Skills**:  
  - “어떻게 일할지”를 패턴으로 묶는 것.  
  - 예: 매번 같은 형식으로 로그 분석, PR 설명, 배포 체크리스트 돌리기.

- **Tool Use**:  
  - “무엇을 이용해 실행할지”.  
  - 예: DB 조회, 외부 API, 사내 시스템, 웹 검색 같은 외부 자원 연결.

그래서 기준을 한 줄로 요약하면,

- 내부 프로세스를 재사용하고 싶다 → Skill  
- 외부 시스템·데이터를 가져와야 한다 → Tool/MCP

## 5. 개인적으로 느낀 점

이번 주 내용은 “에이전트를 더 많이 쓰는 법”이 아니라  
**“에이전트가 일하기 좋은 구조를 만드는 법”**에 가깝다는 느낌이었다.

앞으로는 새로운 도구를 얼마나 많이 아느냐보다,  
새 도구·워크플로를 **얼마나 빨리 흡수해서 내 작업 구조 안에 정리하느냐**가 더 중요한 능력이 될 것 같다고 느꼈다.
