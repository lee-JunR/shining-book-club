# 클로드 코드 작업 시스템 설계 - 하네스 엔지니어링

## 하네스 엔지니어링이란

하네스 엔지니어링(Harness Engineering)은 AI가 단순히 답변을 생성하는 것이 아니라, 안전하고 예측 가능한 방식으로 작업을 수행할 수 있도록 구조를 설계하는 방법론이다.

GUI 기반 AI는 대부분 코드 생성만 수행하고 검증과 판단은 사용자가 직접 담당한다. 반면 CLI 기반 AI는 작업을 이어서 수행할 수 있기 때문에, 사용자가 수행하던 검수와 판단 과정을 절차화하여 시스템 내부에 포함시킬 수 있다.

즉, 하네스 엔지니어링의 목적은 다음과 같다.

* AI가 잘못된 방향으로 작업하지 않도록 제어한다.
* 문제 분석부터 검증까지 작업 과정을 구조화한다.
* 반복 가능한 검수 절차를 시스템화한다.
* 사람이 직접 수행하던 유지보수 업무를 AI가 수행할 수 있도록 만든다.

---

# 클로드 코드의 작업 루프

클로드 코드는 단순히 질문에 답변하는 시스템이 아니라 작업을 수행하는 시스템이다.

작업은 다음 루프를 반복한다.

```text
관찰
↓
판단
↓
실행
↓
결과 반영
↓
재판단
```

따라서 답변 시스템과 작업 시스템을 분리해서 설계해야 한다.

---

# 하네스 설계의 핵심 요소

## 1. 컨텍스트 하네스

AI가 문제를 올바르게 이해하고 필요한 정보만 볼 수 있도록 설계하는 영역이다.

컨텍스트가 부족하면 문제를 제대로 이해하지 못한다.

반대로 컨텍스트가 과도하면 중요하지 않은 정보에 영향을 받아 잘못된 방향으로 판단할 수 있다.

목표는 다음과 같다.

* 문제와 관련된 파일만 탐색하도록 유도
* 수정 범위를 제한
* 로그 분석 지원
* 문제 원인 추론 지원
* 유지보수 관점에서 필요한 정보 제공

---

## 2. 행동 반경 설계

AI가 어떤 행동까지 수행할 수 있는지 정의한다.

핵심 질문은 다음과 같다.

> 이 행동이 실패했을 때 비용은 얼마인가?

실패 비용이 큰 작업은 사람의 승인 또는 추가 검증이 필요하다.

따라서 작업별로 다음을 판단해야 한다.

* AI가 자동 수행 가능한가
* 사람의 개입이 필요한가
* 실행 전에 검증이 필요한가

---

## 3. 작업 검증 레이어

코드 수정은 테스트 통과만으로 충분하지 않다.

무엇을 보장하는지까지 검증해야 한다.

### 자동 검증

반복 가능한 검증 작업

예시

* 테스트 실행
* 린트 검사
* 빌드 확인

### 구조 검증

수정 범위가 예상과 일치하는지 확인

예시

* 불필요한 리팩토링 여부
* 의도하지 않은 파일 수정 여부
* 영향 범위 확인

### 반례 검증

수정이 실패할 수 있는 상황을 검토

예시

* 엣지 케이스
* 예외 상황
* 회귀 테스트

---

## 4. 개입 지점 설계

AI 작업 루프의 어느 지점에 개입할 것인지 결정한다.

중요한 관점은 세 가지다.

### 모델이 무엇을 볼 것인가

Context

### 모델이 무엇을 호출할 수 있는가

Tools / MCP

### 실행 전후에 무엇을 강제할 것인가

Hooks

# Built-in / Skills / MCP / Hooks 역할

## Built-in

클로드가 기본적으로 제공하는 기능

## Skills

반복적으로 사용하는 작업 절차

예시

* 로그 분석
* PR 설명 작성
* 배포 전 체크리스트
* 코드 리뷰 규칙

## MCP

외부 시스템 접근

예시

* Jira
* GitHub
* DB 조회
* 사내 시스템

## Hooks

실행 전후 강제 규칙

예시

* 테스트 필수 실행
* 특정 파일 수정 금지
* PR 생성 전 검증

# AI가 해결 가능한 문제 형태로 변환하기

많은 경우 사용자는 로그만 전달한다.

하지만 로그 자체는 문제 해결에 필요한 정보가 아니다.

AI가 해결 가능한 문제 형태로 구조화해야 한다.

## Symptom

무엇이 언제 실패했는가

## Evidence

실패를 증명하는 자료

예시

* Stack Trace
* Error Log
* Failed Test
* Environment Diff

## Hypotheses

원인 후보를 먼저 정리

예시

* 원인 후보 1
* 원인 후보 2
* 원인 후보 3

수정 전에 후보를 3개 이하로 좁힌다.

## Verification Loop

가설 검증 절차

예시

* 재현 명령
* 검증 테스트
* 반례 테스트
* 롤백 계획

# 작업 시스템의 4단계 구조

## 왜 4단계로 분리하는가

문제를 분석하지 않고 바로 수정하면 다음 문제가 발생한다.

* 잘못된 파일 수정
* 수정 범위 확대
* 연쇄 오류 발생
* 유지보수 비용 증가

따라서 수정 전에 Read Only 기반 분석 과정을 먼저 수행해야 한다.

---

# 1. 규칙 (Rules)

프로젝트의 기본 축을 담당한다.

예시

* CLAUDE.md
* 코딩 스타일
* 금지 사항
* 프로젝트 규칙

---

# 2. 컨텍스트 (Context)

문제 이해를 담당한다.

예시

* 문제 분석
* 관련 파일 탐색
* 수정 범위 지정
* 로그 분석
* 원인 추론

---

# 3. 전략 (Strategy)

어떻게 수정할 것인지 결정한다.

수정 범위와 작업 복잡도에 따라 전략을 선택한다.

### Explore

문제가 단순한 경우

### Plan First

수정 범위가 넓은 경우

### Sub Agent

작업 종류가 다양한 경우

### Thinking Mode

품질이 중요하지만 범위가 크지 않은 경우

---

# 4. 실행 및 검증 (Execution & Validation)

전략을 실제 코드에 적용하는 단계

수행 내용

* 코드 수정
* 테스트 수행
* 검증 수행
* 결과 정리

---

# 정리

클로드 코드의 핵심은 코드를 작성하는 것이 아니다.

핵심은 사람이 수행하던 판단과 검수 과정을 구조화하여 AI 작업 루프 내부에 포함시키는 것이다.

하네스 엔지니어링은 이를 가능하게 만드는 설계 방법이며, 규칙 → 컨텍스트 → 전략 → 실행/검증의 구조를 통해 AI가 안정적으로 작업할 수 있는 환경을 구축하는 것이 목표다.
