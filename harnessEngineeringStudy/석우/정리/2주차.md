# Claude Code 사용하면서 정리한 내용

## 1. `claude.md`는 프로젝트의 규칙을 정의하는 문서

`claude.md`는 Claude Code가 프로젝트에서 항상 참고하는 가이드 문서다.

주로 아래 내용을 작성한다.

* 프로젝트 코딩 컨벤션
* 아키텍처 규칙
* 수정하면 안 되는 영역
* 테스트 실행 방법
* 작업 완료 기준

반면, 민감한 파일 접근을 막거나 특정 디렉터리를 수정하지 못하게 하는 것은 `claude.md`가 아니라 Harness 같은 환경 설정이 담당한다.

> **정리**
>
> * `claude.md` → 프로젝트 규칙
> * Harness → 접근 제어 및 Guardrail

---

## 2. 메모리는 계층적으로 관리된다

Claude의 메모리는 크게 네 단계로 구분된다.

| 계층         | 역할                        |
| ---------- | ------------------------- |
| Enterprise | 조직 전체 정책                  |
| User       | 개인 선호 및 개발 스타일            |
| Project    | 프로젝트 공통 규칙                |
| Local      | 현재 Worktree에서만 사용하는 임시 메모 |

### 메모리 활용

* Project 메모리는 팀 전체가 공유하는 규칙에 적합하다.
* Local 메모리는 실험적인 작업이나 임시 설정에 활용한다.
* Auto Memory는 같은 Git 저장소 내 Worktree 간 공유될 수 있으므로 필요한 경우 상태를 직접 관리하는 것이 좋다.

---

## 3. 세션이 길어질수록 컨텍스트가 오염될 수 있다

대화가 길어질수록 이전 작업 내용이 계속 누적되면서

* 수정 범위가 불필요하게 커지고
* 기존 규칙을 잊거나
* 금지된 파일을 수정하는 경우가 생길 수 있다.

그래서 작업 단위를 명확하게 분리하는 것이 중요하다.

### 자주 사용하는 명령어

#### `/clear`

현재 대화를 초기화하고 새로운 작업을 시작한다.

작업 주제가 바뀌었거나 컨텍스트가 꼬였을 때 사용한다.

#### `/compact`

기존 대화를 요약해서 핵심 내용만 유지한다.

사용할 때는 아래 내용을 함께 포함하도록 지시하면 좋다.

* 변경한 파일
* 실행한 명령어
* 테스트 결과
* 남은 작업

새롭게 정리된 규칙이 있다면 `claude.md`로 옮겨두는 것이 좋다.

---

## 4. 좋은 프롬프트는 범위를 명확하게 정의한다

AI가 엉뚱한 파일을 수정하는 경우는 대부분 지시가 모호해서 발생한다.

작업을 요청할 때는 아래 내용을 함께 작성하는 것이 좋다.

### 1) 해결할 문제

무엇을 해결해야 하는지 한 문장으로 정의한다.

### 2) 수정 가능 범위

어디까지 수정 가능한지 명시한다.

예)

* `service` 패키지만 수정
* Controller는 수정 금지

### 3) 반드시 지켜야 하는 조건

예)

* 기존 API 응답 유지
* 테스트 모두 통과
* DB 스키마 변경 금지

### 4) 완료 기준

작업이 끝났는지 판단할 수 있는 기준을 작성한다.

---

### 큰 작업이라면

바로 구현을 요청하기보다

> 먼저 작업 계획을 작성해줘.

라고 요청한 뒤 계획을 확인하고 구현을 진행하는 것이 안전하다.

---

## 5. 자동화는 구현까지, 최종 판단은 사람이 한다

Claude Code는 반복 작업을 자동화하는 데 적합하다.

예를 들어

1. Issue 분석
2. 작업 브랜치 생성
3. 코드 작성
4. Commit 생성
5. Draft PR 생성

까지는 자동화할 수 있다.

하지만 아래 과정은 반드시 사람이 확인해야 한다.

* 코드 리뷰
* CI 통과 여부
* Merge 승인

AI는 개발을 보조하는 도구이고, 최종 품질은 사람이 책임진다.

---

## 6. 병렬 작업은 Worktree를 사용하는 것이 좋다

같은 디렉터리에서 여러 Claude 세션을 실행하면

* 파일 충돌
* 테스트 충돌
* 컨텍스트 혼선

이 발생할 수 있다.

브랜치마다 별도의 Worktree를 생성하면

* 파일 시스템이 분리되고
* 브랜치도 독립적으로 관리되어
* 여러 작업을 동시에 진행하기 편하다.

작업이 끝난 Worktree는 정리하는 습관을 들이는 것이 좋다.

---

## 7. Merge Conflict는 바로 수정하지 않는다

충돌이 발생하면 바로 코드를 수정하도록 요청하지 않는다.

순서는 다음과 같이 진행하는 것이 좋다.

1. 충돌 원인 분석
2. 양쪽 변경 사항 비교
3. 병합 계획 수립
4. 코드 수정
5. 테스트 수행

충돌 원인을 먼저 이해하고 수정해야 불필요한 코드 손실을 줄일 수 있다.

---

# 내가 가져갈 핵심

* `claude.md`는 프로젝트 규칙을 고정하는 문서다.
* 세션이 길어질수록 컨텍스트가 오염될 수 있으므로 작업 단위를 작게 가져간다.
* 프롬프트에는 **수정 범위, 금지 범위, 완료 기준**을 명확하게 작성한다.
* 큰 작업은 **계획 → 구현** 순서로 진행한다.
* Worktree를 활용하면 병렬 작업을 안전하게 진행할 수 있다.
* AI는 구현을 도와주는 도구이며, 최종 리뷰와 Merge는 사람이 수행한다.
