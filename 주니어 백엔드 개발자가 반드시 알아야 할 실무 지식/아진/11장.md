## MVC 패턴
- 애플리케이션을 역할에 따라 분리하는 대표적인 설계 패턴

### MVC 구성 요소
1. 모델(Model)
  - 비즈니스 로직 담당
  - 예: 회원가입, 암호 변경, 주문 처리
2. 뷰(View)
  - 사용자에게 보여줄 결과 생성
  - 화면 또는 응답 데이터(JSON 등)
3. 컨트롤러(Controller)
  - 사용자 입력 처리
  - 요청 흐름 제어
  - 모델과 뷰 연결

### MVC 패턴의 핵심
- 비즈니스 로직(모델)과 결과 표현(뷰)을 분리
- 요청 처리와 흐름 제어는 컨트롤러에 집중

## 계층형 아키텍처
- 역할에 따라 코드를 계층으로 분리하는 구조
- 대표적인 계층
  - 프레젠테이션 계층
  - 비즈니스 계층
  - 데이터 접근 계층
- 각 계층은 자신의 역할에만 집중
- 상위 계층은 하위 계층에 의존

## DDD(Domain-Driven Design)와 전술 패턴
- 도메인을 중심으로 복잡한 비즈니스 로직을 모델링하는 설계 방식

### 도메인 모델 구성 요소
- 엔티티(Entity)
  - 식별자를 가지며 상태가 변할 수 있음
- 밸류(Value)
  - 값 자체가 중요, 불변
- 애그리거트(Aggregate)
  - 엔티티와 밸류의 묶음
  - 일관성 단위
- 리포지토리(Repository)
  - 애그리거트를 저장하고 조회
- 도메인 서비스(Domain Service)
  - 특정 엔티티에 속하지 않는 도메인 로직
- 도메인 이벤트(Domain Event)
  - 도메인에서 발생한 중요한 사건

## 마이크로서비스 아키텍처
- 서비스를 작은 단위로 분리하여 구성하는 아키텍처

### 마이크로서비스 핵심 개념
1. 독립적 배포
  - 다른 서비스와 독립적으로 변경·배포 가능
2. 도메인 중심 설계
  - 도메인 기준으로 서비스 분리
3. 자신의 상태를 가짐
  - DB를 공유하지 않음
4. 서비스 크기
  - 명확한 기준은 없음
5. 유연함
  - 비용을 들여 확장성과 견고함 확보
6. 아키텍처와 조직의 정렬
  - 조직 구조가 시스템 구조에 영향을 줌

## 이벤트 기반 아키텍처
- 시스템 간 통신을 이벤트로 처리하는 구조

### 구성 요소
1. 이벤트 생산자
2. 이벤트 소비자
3. 이벤트 브로커(또는 라우터)

### 특징
- 생산자와 소비자가 직접 연결되지 않음
- 브로커를 통한 간접 연결
- 독립적 배포 가능
- 새로운 소비자 추가 용이

---

## CQRS 패턴
- CQRS(Command Query Responsibility Segregation)
- 상태 변경과 조회를 위한 모델을 분리

### 특징
- 명령(Command): 시스템 상태 변경
- 조회(Query): 데이터 조회

### 장점
- 명령 모델과 조회 모델을 목적에 맞게 설계 가능
- 조회 성능 최적화에 유리
  - 캐시 적용
  - 조회 전용 DB 확장

### 단점
- 모델이 분리되어 코드량 증가
- 구현 복잡도 상승
