IO 병목 해결 방법
1. 네트워크 IO와 자원 효율의 이해
   서버 프로그램은 본질적으로 네트워크 프로그램입니다. 클라이언트, DB, 레디스, 외부 API 등과 네트워크를 통해 데이터를 주고받습니다.
   블로킹 IO의 문제점
   데이터 입출력 과정에서 스레드는 작업이 완료될 때까지 대기(블로킹) 합니다.
   스레드 실행 → write() 호출 → 대기... → read() 호출 → 대기... → 실행 재개

💡 실제 사례: 전체 실행시간 800ms 중 CPU 사용 시간은 4ms에 불과했습니다. 전체의 99.5%가 IO 대기에 사용된 것입니다.

트래픽 증가 시 발생하는 문제

IO 대기와 컨텍스트 스위칭으로 인한 CPU 낭비
요청마다 스레드 할당으로 인한 높은 메모리 사용량

스레드 1개당 수백 KB~수 MB 메모리를 사용하므로, 1만 명 동시 접속 시 스레드만으로 약 10GB 메모리가 필요합니다.

2. 해결 방법 개요
   방법설명적용 난이도수평/수직 확장서버 추가 또는 자원 증설쉬움 (비용 증가)가상 스레드경량 스레드 사용중간논블로킹 IOIO 대기 없이 처리어려움

3. 가상 스레드 (Virtual Thread)
   개념
   JVM이 관리하는 경량 스레드로, OS 스레드가 아닌 런타임이 스케줄링합니다.
   JVM 스케줄러
   ├── 플랫폼 스레드 1 ──→ 가상 스레드 1, 2
   ├── 플랫폼 스레드 2 ──→ 가상 스레드 3, 4
   └── 플랫폼 스레드 3 ──→ 가상 스레드 5
   플랫폼 스레드 vs 가상 스레드
   항목플랫폼 스레드가상 스레드메모리 (1만 개)약 9.8GB약 28MB생성 시간 (10만 개)약 21초약 0.2초생성 가능 수제한적수십만~백만 개
   동작 방식
   가상 스레드가 블로킹되면 플랫폼 스레드에서 언마운트되고, 플랫폼 스레드는 다른 가상 스레드를 실행합니다.
   가상 스레드 1: 실행 → IO 대기 ────────────────→ 실행 재개
   ↓
   플랫폼 스레드:        가상 스레드 2 실행 → 가상 스레드 3 실행
   주의사항

IO 중심 작업에서만 효과적 (CPU 중심 작업은 효과 없음)
플랫폼 스레드 수보다 가상 스레드가 많아야 이점 발생
synchronized 사용 시 플랫폼 스레드도 함께 블로킹됨 (고정 현상)

가상 스레드의 장점
기존 코드를 크게 수정할 필요 없이 적용 가능합니다. 스프링 프레임워크, MySQL JDBC 드라이버 등 주요 라이브러리가 이미 지원합니다.

4. 논블로킹 IO
   개념
   입출력이 완료될 때까지 스레드가 대기하지 않는 방식입니다.
   java// 블로킹 IO: 데이터 올 때까지 대기
   int bytes = inputStream.read(buffer);

// 논블로킹 IO: 즉시 리턴 (데이터 없으면 0 리턴)
int bytes = channel.read(buffer);
동작 방식 (Selector 사용)
javawhile (true) {
selector.select();  // 가능한 IO 연산이 있을 때까지 대기
for (SelectionKey key : selector.selectedKeys()) {
if (key.isAcceptable()) {
// 클라이언트 연결 처리
} else if (key.isReadable()) {
// 데이터 읽기 처리
}
}
}
```

### 블로킹 IO vs 논블로킹 IO

| 항목 | 블로킹 IO | 논블로킹 IO |
|------|-----------|-------------|
| 스레드 수 | 클라이언트 수만큼 | 소수의 고정 스레드 |
| 메모리 효율 | 낮음 | 높음 |
| 구현 복잡도 | 간단 | 복잡 |

### 리액터 패턴

논블로킹 IO 구현에 사용되는 대표적인 패턴입니다.
```
이벤트 루프 (리액터)
↓
이벤트 발생 대기 → 이벤트 감지 → 핸들러에 전달 → 처리
↑_______________________________________________|
Nginx, Netty, Node.js 등이 이 패턴을 사용합니다.
프레임워크 활용
논블로킹 IO를 직접 구현하기보다 프레임워크 사용을 권장합니다.
java// Reactor Netty 예시
TcpServer.create()
.port(7031)
.handle((in, out) -> {
return in.receive()
.asString()
.flatMap(line -> out.sendString(Mono.just(line)));
})
.bindNow();
```

### 성능 비교 결과

| 구현 방식 | 최대 동접수 (힙 1.5GB) |
|-----------|------------------------|
| 블로킹 IO | 약 6,000 |
| 논블로킹 IO | 약 120,000 |

**약 20배 성능 향상**을 확인했습니다.

## 5. 언제 어떤 방법을 선택할까?

### 의사결정 흐름
```
성능 문제가 있는가?
├── No → 변경 불필요
└── Yes → 네트워크 IO 관련 문제인가?
├── No → DB 최적화, 캐시 등 다른 방법 검토
└── Yes → 구현 변경이 가능한가?
├── No → 서버 확장으로 임시 해결
└── Yes → 가상 스레드 또는 논블로킹 IO 적용
핵심 원칙

문제가 없으면 변경하지 말 것 - 성능 문제가 없는데 복잡하게 구현하지 말자
원인을 정확히 파악할 것 - CPU 중심 작업에는 효과가 없음
현실적 제약 고려 - 우선순위, 팀의 기술 숙련도 등

권장 접근법
상황권장 방법트래픽 증가 예상, 기존 코드 유지 원함가상 스레드극한의 성능이 필요, 복잡도 감수 가능논블로킹 IO당장 문제 해결 필요, 시간 부족서버 수평/수직 확장

💡 팁: 팀 내 기술 탐구 시간을 마련해 필요한 역량을 미리 익혀두면, 문제 발생 시 신속하게 대응할 수 있습니다.
