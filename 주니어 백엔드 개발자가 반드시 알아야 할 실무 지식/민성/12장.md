1. CQRS 패턴 (Command Query Responsibility Segregation)
   개념

명령(Command)과 조회(Query)의 책임 분리: 시스템의 상태를 변경하는 모델(Create, Update, Delete)과 데이터를 조회하는 모델(Read)을 분리하여 구현하는 패턴입니다.

배경: 복잡한 도메인에서는 명령 시 필요한 데이터와 조회 시 필요한 데이터가 달라지는데, 이를 단일 모델로 처리하면 유지보수가 어렵고 성능 최적화가 힘듭니다.

장점

최적화: 각 기능에 맞는 별도 모델 구현이 가능하여, 상호 영향 없이 로직을 짤 수 있습니다.

성능 향상: 조회 모델에만 캐시를 적용하거나, 조회 전용 DB(Read Replica)를 구축하여 성능을 극대화할 수 있습니다.

단점

구현 비용: 모델을 따로 만들어야 하므로 코드가 늘어나고 구조가 복잡해집니다.

데이터 동기화: 명령 DB와 조회 DB가 다를 경우, 데이터 동기화(Eventual Consistency)를 위한 별도 메커니즘(메시징 큐 등)이 필요합니다.

2. 성능 테스트 기초 (Appendix A)
   테스트 종류

부하(Load) 테스트: 예상 트래픽에서 시스템 동작 및 병목 확인.

스트레스(Stress) 테스트: 한계점 돌파 시 시스템의 최대 성능 및 복구 능력 확인.

지속 부하(Soak) 테스트: 장시간 부하를 주어 메모리 누수 등을 확인.

스파이크(Spike) 테스트: 갑작스러운 트래픽 급증 시 안정성 확인.

주요 개념

포화점 (Saturation Point): 처리량이 더 이상 증가하지 않고 성능 저하가 시작되는 최대 성능 지점.

버클 존 (Buckle Zone): 포화점을 넘어 시스템이 한계를 초과해 급격히 무너지는 구간.

주요 측정 지표

응답 시간 (Response Time): 평균값뿐만 아니라 최대값, 95/99 백분위 값을 확인하여 튀는(Outlier) 데이터를 점검해야 함.

처리량 (TPS): 초당 처리 건수.

에러율 & 리소스 사용량: CPU, 메모리 사용량 및 에러 발생 비율 모니터링.

주의 사항

환경 분리: 부하 발생기(Client)와 테스트 대상(Server)은 반드시 다른 장비여야 합니다.

네트워크: 운영 환경과 동일한 네트워크에서 테스트 시 실제 서비스에 영향을 줄 수 있으므로 주의해야 합니다.

외부 연동: 외부 API 등은 Mocking(가짜) 처리하여 외부 시스템 장애를 방지해야 합니다.

설정값: OS의 파일 디스크립터 제한, Nginx 설정, DB 커넥션 풀 등을 테스트 규모에 맞게 미리 조정해야 합니다.

도구: nGrinder(Java기반, UI제공), k6(Go기반, JS스크립트, 가벼움) 등이 자주 사용됩니다.

3. NoSQL 이해하기 (Appendix B)
   특징 및 도입 이유

Not Only SQL: SQL뿐만 아니라 다양한 방식으로 데이터를 처리한다는 의미.

수평 확장(Scale-out): RDBMS(수직 확장 위주)와 달리, 데이터가 늘어나면 서버를 추가하여 용량을 늘리는 데 특화되어 있습니다 (대용량 데이터 처리에 유리).

유연성: 고정된 스키마(Schema-less)가 없어 데이터 구조 변경이 자유롭습니다.

고성능: 트랜잭션의 엄격한 일관성(ACID) 일부를 포기하는 대신 빠른 읽기/쓰기 성능을 제공합니다.

주요 종류

키-값 (Key-Value): 가장 단순하고 빠름. 세션 저장, 캐싱, 설정 관리에 적합 (예: Redis, DynamoDB).

문서 (Document): JSON 유사 형식 저장. 계층적 데이터 표현에 유리 (예: MongoDB).

컬럼 패밀리: 대량 데이터의 쓰기/읽기 성능 최적화 (예: Cassandra, HBase).

그래프: 데이터 간의 복잡한 관계 분석에 유리 (예: Neo4j).
