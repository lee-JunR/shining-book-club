11장. 자주 쓰는 서버 구조와 설계 패턴
서버 애플리케이션 개발 시 아키텍처 수준에서 자주 반복되는 6가지 핵심 패턴입니다.

1. MVC 패턴 (Model-View-Controller)
   사용자 인터페이스와 비즈니스 로직을 분리하여 유지보수성을 높이는 가장 기본적인 패턴입니다. (Spring, Express.js 등)

구성 요소

모델 (Model): 비즈니스 로직 처리 (회원가입, 주문 등).

뷰 (View): 사용자에게 보여질 결과 생성 (HTML, JSON 등).

컨트롤러 (Controller): 사용자 요청 해석, 흐름 제어, 모델 위임, 뷰 선택.

핵심: 모델과 뷰는 서로 의존하지 않으며, 컨트롤러가 이들을 연결합니다. 로직과 UI가 분리되어 영향 범위를 최소화합니다.

2. 계층형 아키텍처 (Layered Architecture)
   각 계층이 특정 역할을 수행하고, 하위 계층에만 의존하는 구조입니다.

일반적인 4계층

표현 (Presentation/UI): 사용자 요청 처리 및 응답 (Controller).

응용 (Application): 실제 요청 처리, 서비스 로직.

도메인 (Domain/Model): 핵심 비즈니스 로직 및 규칙.

인프라 (Infrastructure/Persistence): DB 연동, 외부 통신 (DAO).

주의사항 (흩어지는 도메인 로직): 도메인 계층이 빈약하면 로직이 SQL 쿼리나 컨트롤러로 분산되어 유지보수가 어려워집니다.

3. DDD와 전술 패턴 (Domain-Driven Design)
   복잡한 도메인 로직을 효과적으로 관리하기 위해 도메인 영역에 로직을 집중시키는 설계 방식입니다.

주요 구성 요소

엔티티 (Entity): 고유 식별자(ID)를 가진 객체.

밸류 (Value): 식별자 없이 값 자체를 표현 (불변 권장).

애그리거트 (Aggregate): 연관된 객체(엔티티+밸류)를 묶은 일관성 관리 단위.

리포지토리 (Repository): 도메인 객체의 저장/조회 인터페이스.

도메인 서비스: 엔티티/밸류에 넣기 애매한 로직 구현.

도메인 이벤트: 상태 변경 시 발생하는 이벤트 (타 시스템 알림용).

4. 마이크로서비스 아키텍처 (MSA)
   애플리케이션을 작은 서비스 단위로 분리하여 연동하는 구조입니다.

모놀리식 vs MSA

모놀리식: 배포/관리가 단순하나, 규모가 커지면 빌드/배포 속도 저하 및 부분 장애가 전체로 확산될 위험이 있음.

MSA: 독립적 배포가 가능하고 확장성이 좋으나, 인프라 복잡도가 높고 관리가 어려움.

핵심 개념: 독립적 배포(가장 중요), 도메인 중심 모델링, DB 공유 금지(API 통신).

5. 이벤트 기반 아키텍처 (Event-Driven Architecture)
   시스템 간 통신을 '이벤트(과거에 발생한 사실)'를 통해 처리하는 구조입니다.

구성: 이벤트 생산자 → 이벤트 브로커(Kafka 등) → 이벤트 소비자.

장점: 시스템 간 결합도를 낮추어 독립적 배포 및 확장이 용이함.

활용: 결제 완료 후 배송 시스템 트리거, 이상 감지 알림 등.

6. CQRS 패턴 (Command Query Responsibility Segregation)
   명령(상태 변경) 모델과 조회(데이터 읽기) 모델을 분리하는 패턴입니다.

이유: 복잡한 시스템에서는 변경에 필요한 데이터와 조회에 필요한 데이터가 다르기 때문.

장점: 각 기능에 최적화된 모델 구현 가능, 조회 성능 튜닝(캐시, 전용 DB) 용이.

단점: 구현 복잡도 증가, 데이터 동기화 이슈 발생 가능.

1. 성능 테스트 종류
   시스템이 트래픽을 얼마나 감당할 수 있는지 측정합니다.

종류 | 설명 |        목적             |
|---|:---:|----------------------:|
| `부하 (Load)` | 예상 부하 수준에서 동작 확인 | `"응답 시간, 처리량, 병목 파악"` |
| `스트레스 (Stress)` | ,한계 이상의 부하 부여 |  `"최대 성능 및 복구 능력 확인"` |
| `지속 부하 (Soak)` | 장시간 부하 유지 |   `"메모리 누수 등 안정성 검증"` |
| `스파이크 (Spike)` | 급격한 트래픽 변화 부여|      `"반응성 및 안정성 검증"` |

---
2. 핵심 용어 및 주의사항
   포화점 (Saturation Point): 처리량이 더 이상 증가하지 않고 성능 저하가 시작되기 직전의 최대 처리량 지점.

버클 존 (Buckle Zone): 시스템 한계를 초과하여 성능(처리량, 응답 속도)이 급격히 무너지는 구간.

주요 병목 지점: DB 쿼리(Slow Query), 외부 API 연동 대기 시간, 커넥션 풀(Connection Pool) 부족 등.
