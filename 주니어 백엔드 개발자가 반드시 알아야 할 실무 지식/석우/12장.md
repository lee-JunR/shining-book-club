## ** 정리 **

#### 처음 해보는 성능 테스트를 위한 기본

```
성능 테스트의 종류는 부하, 스트레스, 지속 부하, 스파이크 테스트로 크게 4종류가 있으며, 특정 작업 부하 상태에서 응답성과 안정성 측면에서 시스템이 어떻게 작동하는지를 확인하기 위해 수행하는 테스트 관행이다.

성능 테스트를 진행하는 일반적인 방식은 낮은 부하에서 시작해서 점진적으로 부하를 높이는 것인데 예를 들어 사용자 수를 100명에서 시작해 200명, 300명 단계적으로 부하를 높여가며 성능을 측정한다.

성능이 저하되기 전의 최대 처리량을 포화점이라고 하고 포화점을 지나 성능이 꺾이기 시작하는 구간을 버클존이라고 한다.

버클이란 버클링(좌굴)은 물리 현상에서 빌려온 개념으로 세로로 긴 구조물을 세로 방향으로 압축할 때 어느 순간 압축 하중을 견디지 못하고 구부러지는 것을 말하며 버클존의 버클은 이 현상에서 빌려온 단어로 시스템이 한계를 초과했을 때 급격하게 성능이 저하되는 상황을 표현한 것이다.

성능 테스트에서 중요한 지표 중 하나는 응답 시간인데, 응답 시간은 다음과 같이 여러 값을 측정한다. (평균, 최대, 최소, 중앙, 99%나 95% 백분위)

처리량 : TPS(초당 트랜잭션 건수)처럼 초 단위로 얼마나 많은 요청을 처리했는지를 나타내며, 테스트를 진행하는 동안 처리량은 변화하므로 최대, 평균, 최소 값을 함께 구한다.

에러율 : 에러율은 전체 요청 중에서 에러가 발생한 비율을 의미하고, 부하 테스트를 하다 보면 처리량은 높은데 에러도 함께 증가하는 상황이 생긴다. 에러가 발생한다는 건 시스템이 부하를 감당하지 못하고 있다는 신호일 수도 있다.

CPU 사용률 : 부하 테스트 중에 CPU 사용률도 함께 모니터링을 해야 하는데, 일반적으로 웹 서버의 성능 문제는 DB나 외부 연동에서 발생하지만 높은 CPU 사용률로 인해 문제가 생기는 경우도 있다.

성능 테스트 설계 시 고려 사항으로는 시스템의 트래픽 패턴, 동시 요청 사용자 수/트래픽 규모, 기능별 요청 비율, 데이터 크기, 워밍업, 적절한 목표치 설정 등 다양하게 있다.
```

#### NoSQL 이해하기

```
NoSQL은 데이터베이스의 전통적인 관계형 데이터베이스보다 덜 제한적인 일관성 모델을 이용하는 데이터의 저장 및 검색을 위한 메커니즘을 제공한다. 이러한 접근에 대한 동기에는 디자인의 단순화, 수평적 확장성, 세세한 통제를 포함하고 NoSQL 데이터베이스는 단순 검색 및 추가 작업을 위한 매우 최적화된 키 값 저장 공간으로 레이턴시와 처리량과 관련하여 상당한 성능 이익을 내는 것이 목적이다.

NoSQL이 사용되는 주된 이유는 대용량 데이터나 분산 처리, 고속의 읽기와 쓰기 성능, 특정한 요구사항에 맞는 데이터 설계, 비정형 데이터 처리 또는 유연한 스키마 등의 이유가 있다.

NoSQL을 나누는 다양한 기준이 존재하는데 일반적으로 다음 4가지 유형으로 나눠진다.
1. 키-값 DB
2. 문서 DB
3. 칼럼 패밀리 DB
4. 그래프 DB
```

#### 결론

이번 장에서는 성능 테스트의 기본 개념과 NoSQL 데이터베이스에 대한 기초적인 이해를 통해, 시스템의 확장성과 안정성을 확보하기 위해 개발자가 반드시 고려해야 할 핵심 요소들을 정리하였다.

성능 테스트는 단순히 시스템의 한계를 측정하는 작업이 아니라, 실제 운영 환경과 유사한 부하 조건에서 응답 시간, 처리량, 에러율, 자원 사용률 등을 종합적으로 분석하여 서비스 품질을 검증하는 중요한 과정임을 알 수 있었다. 특히 포화점과 버클존 개념을 통해 시스템이 어느 시점에서 급격한 성능 저하를 겪는지 이해하는 것은, 적절한 용량 산정과 장애 예방에 큰 도움이 된다.

또한 NoSQL은 관계형 데이터베이스의 한계를 보완하기 위해 등장한 대안으로, 대용량 데이터 처리와 수평적 확장, 고속의 읽기·쓰기 성능이 요구되는 환경에서 강점을 가진다는 점을 확인하였다. 키-값, 문서, 칼럼 패밀리, 그래프 DB와 같이 다양한 유형의 NoSQL은 각각의 특성과 사용 목적이 다르므로, 시스템 요구사항에 맞는 데이터베이스 선택이 중요하다.

결국 성능 테스트와 NoSQL에 대한 이해는 단순한 기술 지식에 그치지 않고, 시스템의 안정성, 확장성, 그리고 장기적인 운영 효율성을 결정짓는 중요한 판단 기준이 된다. 따라서 개발자는 성능 테스트 결과를 기반으로 시스템을 지속적으로 개선하고, 데이터 특성과 트래픽 패턴에 적합한 저장소를 선택함으로써 더 신뢰할 수 있는 서비스를 설계해야 할 것이다.
