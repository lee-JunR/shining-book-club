## ** 자주 쓰는 서버 구조와 설계 패턴 정리 **

### MVC 패턴

MVC는 Model-View-Controller의 약자로 애플리케이션의 구성 요소를 세 가지 역할로 나누는 패턴이다.

모델(Model)은 비즈니스 영역의 로직을 처리하고, 뷰(View)는 사용자가 보게 될 결과를 생성하며, 컨트롤러(Controller)는 사용자의 입력 처리와 흐름 제어를 담당한다.

이 패턴의 핵심은 비즈니스 로직을 처리하는 모델과 결과를 생성하는 뷰를 분리하여 서로의 변화에 영향을 덜 받게 하는 데 있다.

컨트롤러는 사용자의 요청을 해석해 모델에 실행을 위임하고 그 결과를 보여줄 뷰를 선택하는 역할에 집중합니다. 이러한 구조는 역할에 따라 소스코드를 분리함으로써 유지보수를 용이

### 계층형 아키텍처

각 계층이 특정 역할을 수행하고, 오직 하위 계층에만 의존하는 특징을 가진 전통적인 패턴이다.

웹 애플리케이션에서는 일반적으로 사용자와의 상호작용을 담당하는 표현 계층, 로직 처리를 담당하는 응용 계층, 도메인 로직을 구현하는 도메인 계층, 그리고 DB 연동 등을 지원하는 인프라 계층의 4개 계층으로 구성한다.

이 아키텍처는 구조가 단순하고 규칙이 명확하여 코드 실행 흐름을 추적하기 쉽다는 장점이 있습니다.

다만, 도메인 계층이 빈약할 경우 로직이 응용 계층이나 인프라 계층(쿼리 등)으로 분산되어 유지보수가 어려워질 수 있으므로 주의

### DDD와 전술 패턴

도메인 주도 설계(DDD)의 전술 패턴은 복잡한 도메인 로직을 효과적으로 관리하기 위해 엔티티, 밸류, 애그리거트, 리포지토리, 도메인 서비스, 도메인 이벤트와 같은 구성 요소를 사용한다.

애그리거트는 관련된 객체들을 하나로 묶어 일관성을 관리하는 단위가 되며, 이를 통해 도메인 로직의 응집도를 높일 수 있다.

이러한 패턴은 도메인 로직을 애그리거트 단위로 모아 관리함으로써 로직이 여러 곳으로 흩어지는 것을 방지하고 복잡도를 낮춘다.

응용 서비스는 이러한 도메인 모델을 사용해 사용자의 요청을 처리하는 역할을 수행

### 마이크로서비스 아키텍처

마이크로서비스 아키텍처(MSA)는 애플리케이션을 더 작은 서비스 단위로 분리하고 각 서비스가 네트워크를 통해 연동되는 구조이다.

이 구조의 가장 큰 장점은 독립적인 배포가 가능하며, 서비스별로 기술 선택의 유연성을 가질 수 있고 성능 확장이 용이하다는 점이다.

하지만 모놀리식 아키텍처에 비해 인프라가 복잡해지고 테스트나 디버깅이 어려워질 수 있다는 단점도 존재한다.

따라서 단순히 유행을 따르기보다는 도메인 경계를 명확히 설정하고, 조직의 역량과 비용 대비 이점을 충분히 고려하여 도입해야 한다.

### CQRS

CQRS는 Command Query Responsibility Segregation의 약자로, 시스템의 상태를 변경하는 명령(Command) 모델과 상태를 조회하는 조회(Query) 모델을 분리하는 패턴이다.

복잡한 도메인에서는 명령할 때 사용하는 데이터와 조회할 때 필요한 데이터 형식이 서로 다른 경우가 많은데, 이를 하나의 모델로 처리하면 코드가 복잡해지고 유지보수가 어려워진다.

CQRS를 적용하면 각 기능에 맞게 모델을 최적화할 수 있어 조회 성능을 향상시키고 모델의 복잡도를 낮출 수 있다.

다만, 모델을 따로 만들어야 하므로 구현해야 할 코드가 늘어나고 두 모델 간의 데이터 동기화 수단이 추가로 필요할 수 있다.

#### 결론

주니어 백엔드 개발자는 실무에서 자주 사용되는 이러한 서버 구조와 설계 패턴들을 이해하고 상황에 맞게 적용할 수 있어야 한다.

MVC 패턴과 계층형 아키텍처는 코드의 역할을 분리해 유지보수성을 높이는 기초가 되며, DDD의 전술 패턴은 복잡한 로직을 응집도 있게 관리하는 데 도움을 준다.

또한, 시스템의 규모와 복잡도에 따라 마이크로서비스 아키텍처나 CQRS 같은 패턴을 선택하여 독립적인 확장성과 성능 최적화를 꾀할 수 있다.

중요한 것은 각 패턴의 장단점을 명확히 파악하고, 현재 서비스의 요구사항과 조직의 역량에 비추어 가장 합리적인 설계 방안을 선택하는 것이다.
