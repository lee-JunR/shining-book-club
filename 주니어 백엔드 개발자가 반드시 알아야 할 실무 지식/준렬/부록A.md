# 부록 A 처음 해보는 성능테스트를 위한 기본 정리

## 성능 테스트 종류

서버가 어느정도의 트래픽을 감당할 수 있는지 확인하는 것을 성능 테스트라고 한다.

성능 테스트의 종류는 다음과 가탇.

- 부하(load) 테스트 : 특정 예상 부하에서 시스템이 어떻게 동작하늕 확인한다. → 주요 기능의 응답 시간과 처리량 등의 지표를 확인하고 병목을 파악하는데 도움을 준다.
- 스트레스(stress) 테스트 : 시스템의 최대 성능을 확인하기 위한 테스트이다.
  → 예상을 뛰어넘는 부하가 있을 때 시스템이 얼마나 버틸 수 있는지 확인한다.
- 지속 부하(soak) 테스트 : 시스템이 장시간동안 지속적인 부하를 견딜 수 있는지 검증한다.
  → 성능 저하가 발생하는지, 메모리 누수가 있는지 탐지할 수 있다.
- 스파이크 테스트 : 급격하게 트래픽이 변화할 때 반응성과 안정성을 검증한다. 순간적으로 트래픽이 급증할 때 성능저하나 실패가 발생하는지 확인한다.

## 성능 테스트 - 부하테스트

성능 테스트는 보통 낮은 부하를 걸어서 점진적으로 부하를 높이면서 진행된다.

이렇게 부하를 증가시키면 초기에는 **처리량(일정 시간 동안 시스템이 실제로 처리한 작업(요청)의 개수)**도 함께 증가한다.

하지만 부하가 너무 높아지면 처리량의 증가 폭이 줄어들기 시작하고 응답 시간이 급격히 저하된다. 이때 성능이 저하되기 전의 최대 처리량을 **포화점** 이라고 한다.

포화점은 시스템이 감당할수 있는 성능 한계 지점이다.

포화점을 지나 성능이 꺾이는 구간을 버클 존이라고 한다.

### 포화점 체크

포화점이 목표로 한 성능보다 높다면 성능 테스트를 통과할 수 있지만 낮거나 같다면 병목지점을 찾아 제거해야한다.

웹 서비스의 병목 지점은 보통 호출 비중이 높으면서도 응답 시간이 긴 기능과 관련되어있다.

응답 시간에 영향을 주는 요인

- DB 연동
- 외부 서비스 연동 시간
- 트래픽 대비 부족한 커넥션 풀 크기

## 주요 성능 지표

성능 테스트에서 중요한 지표 중 하나는 **응답시간 이다.**

응답시간을 측정하는 지표는 다음과 같다.

1. 평균
2. 최대
3. 최소
4. 중앙
5. 99%, 95% 같은 백분위

응답시간의 평균과 최대 응답시간의 차이가 크다면 백분위 값을 함께 확인해야한다.

이 백분위 값이 최대 응답시간보다 평균값이 가깝다면 최대 값은 이상치일 수 있다.

중앙값(모든 값의 중간과 평균값의 차이도 중요하다. 응답 시간이 중앙값보다 평균 값이 더 큰 좌편향 분포를 보인다면 이는 응답시간이 전반적으로 평균보다 느리다는 것을 의미한다.

- 처리량
  - TPS 처럼 초 단위로 얼마나 많은 요청을 처리할 수 있는지를 나타낸다.
    → 테스트를 진행하는 동안 달라질 수 있으므로 평균, 최소, 최대 값을 함께구한다.
- 에러율
  - 전체 요청중에서 에러가 발생한 비율
- CPU 사용율
  - 일반적으로 웹서버의 문제는 DB나 외부 여농에서 발생ㄷ하지만 높은 CPU 사용률로 인해 문제가 생기는 경우도 있다.

## 테스트 설계 고려 사항

성능 테스트를 설계할 때 고려할 요소는 다음과 같다.

- 시스템의 트래픽 패턴
  - 트래픽은 보통
- 동시 요청 사용자 수 / 트래픽 규모
- 기능별 요청 비율
- 데이터 크기
- 워밍업
- 적절한 목표치 설정

트래픽이 짧은 시간에 증가하는 패턴 → 성능 테스트도 짧은 시간 간격으로 부하를 늘린다.

## 성능 테스트 도구

`nGrinder` : 네이버에서 개발한 부하 테스트 도구. grinder 기반.

1개의 컨트롤러와 다수의 에이전트로 이루어져 있음.

에이전트 : 부하를 발생시키고 테스트 대상 서버의 시스템 지표를 모니터링함.

`k6` : Grafana로 유명한 Grafana labs 에서 만든 부하 테스트 도구.

고 언어로 개발됨. 테스트 스크립트는 자바스크립트.

고루틴을 사용해서 부하를 발생시키기 때문에 더 적은 자원으로 더 많은 부하를 발생시킬 수 있음.

여러 장비에 부하를 주려면 k6 클라우드 사용 ( 유료)

Locust : 파이썬을 사용해서 테스트 정의하여 부하를 발생시킴.

`Gastling` : 자바 스칼라 코틀린 자바 스크립트를 이요해서 테스트 스크립트 작성.

`Jmeter` : 분산환경에서도 부하 발생 가능.

## 실행 주의 사항

성능 테스트를 진행할때에는 테스트 대상과 부하 발생 시스템을 한 장비에서 실행하며 안된다는 것이다.

서버 설정에 제한을 걸고 부하테스트를 하면 안된다. 서버의 스레드 풀 개수나 커넥션 풀의 크기가 작으면 부하 테스트가 제대로 이루어지지 않으니 테스트할 부하 규모에 맞게 풀의 크기를 조정하자.

운영 시스템과 동일한 네트워크에서 테스트를 진행하면 운영 시스템에도 문제가 발생할 수 있으므로 조심해야한다.

외부 서비스를 연동하는 기능을 테스트할때에는 실제 외부 서비스를 연동하면 안된다.

테스트 대상 시스템은 최대한 실제 운영 화녁ㅇ과 동일하게 구성해야한다.
