## MVC 패턴

MVC 패턴은 Model, View, Controller 패턴의 약자로 비즈니스 로직을 처리하는 모델과 결과를 생성하는 뷰를 분리하고 애플리케이션의 흐름 제어나 사용자의 요청 처리는 컨트롤러에 집중하는 패턴이다.

컨트롤러를 통해 화면과 비즈니스 로직을 분리함으로 써 모델과 뷰는 서로의 변화에 영향을 덜 받게하여 유지보수를 용이하게 만든다.

## 계층형 ㅏ키텍쳐

계층형 아키텍쳐는 각 계층마다 특정한 역할을 수행하고 하위 계층에만 의존하는 아키텍처를 말한다.

하위계층은 상위계층에 대한 의존을 갖지 않는다.

웹 애플리케이션은 보통 4개 계층을 구성한다.

표현(UI) → 응용 → 도메인(모델) → 인프라(영속)

- 표현 계층 : 사용자와의 상호 작용을 담당
- 응용 계층 : 모델이나 인프라 계층에서 필요한 계층을 구현하고 결과를 표현 계층에 리턴
- 도메인 : 상태변광과 같은 로직
- 인프라 영속계층 : DB 연동이나 문자 발송과 같은 기현 기술을 지원

→ 도메인 계층이 없으면 도메인 로직이 각 계층으로 퍼지는 영향을 주어 유지보수가 힘들어진다.

## DDD 와 전술 패턴

도메인 로직이 다른 계층으로 퍼지지 않도록 하는 디자인 패턴이다.

### 구성 요소

- 엔티티 : 고유의 식별자를 가지며 각 엔티티는 식별자로 구분된다. 내부 상태가 변경되어도 식별자는 바뀌지 않는다.
- Value :고유의 식별자를 갖지 않으며 개념적인 값을 표현한다. 금액, 배송 주소와 같은 값들이 value 가 된다. 값은 불변으로 구현하는 것을 추천한다.
- Aggregate : 관련된 객체를 묶어 하나의 개념적인 단위를 표현한다. 예를 들어 주문 애그리거트는 Order엔티티, OderLine 밸류 집합, ShippingAddress 밸류로 굿성될 수 있다. 모델의 일관성을 관리하는 단위가 된다.
- Repository : 도메인 객체를 물리적인 저장소와 연결할 때 사용되는 모델이 리포지토리. repogitory는 애그리거트 단위로 존재한다
- 도메인 서비스 : 특정한 애그리거트에 속하지 않ㄷ은 로직을 구현한다. 외부의 연동이 필요한 도메인 로직도 도메인 서비스로 사용해서 표현한다.
- 도메인 이벤트 : 도메인 내에서 발생한 이벤트를 표현한다. 도메인의 상태가 변경될 때 도메인 이벤트가 발생한다. 도메인 이벤트는 주로 다른 부분에 변화를 알리기 위해 사용된다.

## 마이크로 서비스 아키텍쳐

작은 단위로 서비스를 분리하고 각 서비스가 연동되는 구조를 갖는 아키텍쳐.

- 독립적, 지속적 배포가 용이하다.
- 성능 확장이 용이하다.
- 기술 유연성 확보 가능

단점

- 테스트와 디버깅이 어려움
- 인프라가 복잡함
- 소통 부하
- 무분별한 서비스 생성은 분산 모놀리식이 될 수 있다.

## 모놀리식 아키텍쳐

하나의 애플리케이션에 모든 과정을 구현한 아키텍쳐

- 배포가 단순하다
- 코드 관리가 쉽다
- 성능을 높이기 위해 복잡한 구조를 가질 필요가 없다.
- 테스트와 디버깅이 쉽다.

단점

- 규모가 커질수록 개발 속도가 느려질 수 있다.
- 한 기능의 문제가 전체에 영향을 준다.
- 구현 기술 변경에 어려움이 잇다.
- 작은 변경도 전체를 다시 배포해야한다.

두가지 아키택쳐 선택 기준

1. 독립적인 배포
2. 도메인 중심 모델링 : 각 마이크로 서비스는 도메인을 기준으로 구분해야한다. 한 도메인의 기능 구현이 여러 마이크로 서비스에 걸쳐있으면 출시 비용이 증가한다.
3. 자신ㅇ늬 상태를 가짐: 마이크로 서비스는 DB를 공유하지 않는다. 다른 마이크로 서비스의 데이터를 사용할 경우 DB 에 직접 접근이 아닌 API 등을 통해 접근한다.

## 이벤트 기반 아키텍쳐

두 시스템간 통신에 이벤트를 사용하는 구조.

이때 이벤트는 과거에 발생한 사실을 의미한다.

그래서 이벤트는 보통 인증에 실패함 과같은 과거형으로 쓰인다.

### 이벤트 기반 아키텍쳐의 구성요소

1. 이벤트 생산자
   1. 이벤트를 생산해서 브로커에 전달한다.
2. 이벤트 소비자]
   1. 이벤트를 받아 적절하게 반응한다.
3. 이벤트 브로커(또는 라우터)
   1. 해당 이벤트를 관심있어하는 소비자에게 전달한다.
   2. 보통 메시징 기술을 사용하는데 카프카가 바로 이벤트 브로커로 많이 활용된다.

## CQRS 패턴

아주 단순한 시스템은 상태 변경과 상태 조회가 동일한 모습을 갖지만 조금만 복잡해져도 상태 변경에 사용되는 데이터와 상태 조회 시 사용하는 데이터에 차이가 발생한다.

명령과 조회에 다른 데이터를 사용하는데도 하나의 모델로 명령과 조회를 구현하면 코드가 복잡해지고 코드 유지보수가 어려워진다. 이때에는 CQRS 패턴을 사용해 단일 모델 사용으로 인해 발생하는 복잡도 문제를 해결할 수 있다.

CQRS는 Command Quert Responsibilty Segregation 의 약자로 조회를 위한 모델과 명령을 위한 모델을 분리하는 패턴이다.

명령은 상태 변경을 조회는 상태 조회를 의미한다.

유지보수성이 좋아지고 DB 확장에 용이하다. 하지만 각 기능마다모델을 따로 만듣ㄹ어야하므로 코드가 늘어난다.

따라서 추가되는 코드 대비 얻는 이점이 더 큰지 따져봐야한다.

단순한 모델의 경우 적용하지 말고 복잡한 경우에는 얻는 이점이 크다.

높은 트래픽을 처리하기 위해 조회 모델을 별 도 조히ㅗ 전용 DB 에 구축할 경우 서로 다른 DB 간의 데이터 동기화를 위해 추가적인 메시징 수단을 도입해야한다.
