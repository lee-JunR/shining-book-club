데이터 요구사항이 다양해지면서 그에 맞춰 데이터 설계 방식도 다양해지고 있다.

- 수천만 회원 간의 연결 관계를 분석해서 친구를 추천한다.
- 대량의 데이터를 실시간으로 수집하고통계를 추출한다.
- 테라바이트에서 페타바이트 이상의 데이터를 처리한다.

RDBMS로도 이런 요구를 처리할 수는 있지만 더 알맞은 기술을 찾아야 할 때도 있다.

과거에는 No-SQL은 관계형 데이터베이스의 기술을 배제한 데이터베이스였지만 시간이 지남에 따라 NoSQL 기술도 SQL 과 유사한 언어를 지원하기 시작했다.

NoSQL 을 사용하는 이유

- 대용량 데이터
  - 수평 확장을 통해 Cassendra나 HBASE의 경우 데이터 저장 용ㅇ량을 늘릴 수 있다.
- 고속 읽기 쓰기 성능
  - 분산처리나 높은 성능과 같은 특징을 제공하기 위해 ACID 요건 중 일부를 지원하지 않는 NoSQL 도 있다.
- 특정한 요구사항에 맞는 설계
- 비정형 데이터 처리 또는 유연한 스키마

수평 확장을 통한 대용량 데이터 처리에 강점이 있는 NoSQL 도 있다.

### NoSQL 종류

NoSQL 을 나누는 다양 한 기준

- 키-값 DB , 예 : 레디스, 다이나믹 DB
  - 세션 관리 : 인증 토큰과 같은 세션 정보 저장
  - 캐시 : 자주 사용하는 데이터 캐싱
  - 설정 관리 : 설정은 키 값의 형태를 가지므로 적합
- 문서 DB , 예 : 몽고 DB
  - 컨텐츠 관리 : 다양한 종류의 컨텐츠 관리
  - 제품 카탈로그 : 다양한 메타데이터를 저장하는 제품 카탈로그
- 칼럼 패밀리 DB
  - 키 - 값 의 확장버전, 예 : Cassandra, Hbase
  - 대규모 데이터 관리 : 채팅 메시지나 IoT 데이터 등 대규모 저장과 조회
- 그래프 DB
  - 노드 데이터를 연결하는 엣지 데이터가 있음. 예 Neo4j
  - 소셜
  - 추천
  - 부정 탐지

## NoSQL 도입 시 고려사항

1. 트랜잭션 지원 여부 고려하기.
   - 다수의 NoSQL이 트랜잭션을 지원하지 않으므로 어플리케이션단에서 따로 진행해야함.
   - RDBMS와 함께 사용할때는 두 데이터 간 동기화도 고려해야함.
2. 데이터 모델이 요구사항에 저갛ㅂ한지 확인해야함.
3. 확장성과 성능 요구
   - 일반적으로 NOSQL이 확장성이 높고 속도가 빠르지만 높은 일관성을 지원하는 RDBMS 와는 달리 궁극적 일반성을 지원함으로 성능보다 일관성이 중요한 경우 이 특징이 요구를 충족하는지 검증해야함
4. 운영과 개발 역량

## CAP 정리

cap 정리란 다음 세 조건을 만족하는 분산 시스템은 존재하지 않는다는 것을 증명한 정리이다.

- 일관성(Consistency) : 모든 노드가 같은 순간에 같은 데이터를 바라볼 수 있다. 한 노드가 변경되면 모든 노드의 데이터도 동일한 값으로 바뀐다.
- 가용성(Availablility) : 모든 요청이 성공 ㄸ는 실패 결과를 반환할 수 있다.
- 분할내성(Partition tolerance) : 네트워크 장애가 발생해도 시스템이 계속 동작 가능하다.

CAP 정리에 따르면 위 조건 중 최대 2가지만 충족 가능하다.

NoSQL은 분산 시스템을 기반으로 하기에 분할내성을 기본으로 한다. 따라서 NoSQL은 가용성이나 일관성을 포기해야한다. 어떤 걸 우선순위에 두냐에 따라 선택할 NoSQL 도 달라진다.
